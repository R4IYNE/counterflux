## Why production LCP (0.7s) beats v1.1 lab LCP (2.49s)

The v1.1 lab measurement ran against `vite preview` on localhost — Vercel's edge CDN doesn't apply, all assets serve from a single localhost connection. Production benefits from:

1. **Vercel edge CDN** — static assets (HTML, JS, CSS, fonts) serve from the nearest geographic node. The Lighthouse run originated from a UK location (Windows + Vercel `dub1` edge per X-Vercel-Id header), giving sub-50ms first-byte for the HTML.
2. **HTTP/3 multiplexed asset delivery** — concurrent chunk fetches without head-of-line blocking.
3. **`Cache-Control: no-cache` + repeat-visit cache hits** — first-visit is uncached, but the hashed JS/CSS chunks return revalidation 304s on subsequent navigation.
4. **The static paint-critical `<h1>` from Phase 13 Plan 5** — auth-wall decorating pre-existing DOM means the LCP candidate is in the initial HTML byte stream.

The 2.49s v1.1 number remains valid as the **lab worst-case ceiling**; the 0.7s production number is the **real-world floor for desktop UK**. Both are useful — the lab number protects against regressions, the production number sets reasonable user expectations.

## Disclaimers

- **Single run, not median-of-3.** v1.1's PERF-BASELINE.md captured a median; this addendum is a single one-shot run for UAT-02 closure. If a future regression emerges, re-run with `numberOfRuns: 3` per the v1.1 methodology.
- **Geographic bias.** Run originated from Windows NT in the UK; Lighthouse + Chrome surfaced from the local machine (HeadlessChrome 147), so network conditions reflect a fast UK broadband connection and the closest Vercel edge. Users on slower networks or further geographies will see higher numbers.
- **First-visit-only.** Lighthouse runs in a fresh browser session. Repeat-visit (warm cache, IndexedDB populated) numbers are not captured here — the v1.1 final Lighthouse run measured this same way.
- **Phase 15 EDHREC + Spellbook proxies are NOT yet in production at measurement time.** The numbers above reflect the v1.1 deployed build (`dfad4e7`), not Phase 15. When `master` is next promoted to production, proxy traffic via `/api/edhrec/*` and `/api/spellbook/*` may add small latency on the first synergy/combo fetch in deck builder workflows. That latency is upstream-bound (json.edhrec.com / backend.commanderspellbook.com), not a Counterflux client-bundle concern, and is not captured by Lighthouse navigation-mode.
