<!-- Hook install order (CRITICAL — hooks must attach AFTER db.open() resolves) -->

`src/db/schema.js` currently declares UUID_TABLES hooks at module top-level. Those fire correctly because they're attached before any write hits. For sync-queue hooks, we want them attached only AFTER the full schema has been migrated and db.open() resolved — otherwise mid-migration writes during v6/v7/v8 upgrades would enqueue sync ops for legacy rows, which is wrong.

Solution: add an `installSyncHooks()` function to sync-engine.js, exported. Call it from a post-migration hook in src/services/migration.js OR from src/main.js after runMigration() resolves. This plan ships it in main.js inside the Alpine.effect that watches auth.status — the hooks install lazily once, the first time auth transitions to 'authed'.

<!-- classifyError mapping (D-10, PITFALLS §9) -->

transient (retry with backoff):
- HTTP 429, 5xx
- Network failures: 'network', 'fetch failed', 'timeout'
- err.code undefined AND err.message matches /network|fetch|timeout/i

permanent (dead-letter):
- HTTP 400, 401, 403, 409, 422
- PostgREST: 'PGRST301' (auth), 'PGRST204' (column missing)
- Postgres SQLSTATE: 42501 (RLS), 22xxx (data exception), 23xxx (integrity violation)
- unknown codes default to 'permanent' (fail-fast)
