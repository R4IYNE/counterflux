# Plan 14-06: Auth-wall stale static element fast-path bug

## Context

Discovered during Plan 14-05 live UAT on 2026-04-26. After the schema-parity migration landed and the user signed in, the app boot got stuck on the bare `COUNTERFLUX` wordmark with no other UI visible — sidebar, topbar, and main route all rendered into the DOM (`bodyHTML` 62 KB, `mainOffsetWidth: 1047`) but invisible because a static element was covering everything.

### Root cause

Phase 13 Plan 5 Task 6 (the LCP optimisation) ships a paint-critical `<div id="cf-auth-wall">` directly in `index.html` so the COUNTERFLUX h1 paints on HTML parse instead of waiting ~6s for the full JS boot chain. `src/components/auth-wall.js` is supposed to detect this static element on `openAuthWall()` and decorate it, then `closeAuthWall()` removes it once auth flips to `'authed'`.

The race: if `auth.status` rehydrates to `'authed'` BEFORE `Alpine.effect()` first invokes `openAuthWall()`, the module-local `wallEl` reference stays `null`. The original `closeAuthWall()` early-returned on `!wallEl`, so the static element was never removed. It stayed pinned at `position: fixed; inset: 0; z-index: 90` covering the entire viewport with the bare h1 visible and nothing else interactive.

This is a v1.1 boot regression. v1.0 didn't ship the static element. Phase 13 added it for LCP without updating `closeAuthWall()` for the rehydrate-fast-path race.

### Fix

Belt-and-braces approach in `closeAuthWall()`: if `wallEl` is set, remove via the existing path (no behaviour change). If `wallEl` is null, also check for and strip a stale `#cf-auth-wall` element from the DOM. Six lines of code, fully backwards-compatible.

    Update `src/components/auth-wall.js` `closeAuthWall()`:

    ```js
    export function closeAuthWall() {
      if (wallEl) {
        wallEl.remove();
        wallEl = null;
        return;
      }
      // Phase 14.06 — fast-path bug fix.
      // index.html ships a paint-critical static <div id="cf-auth-wall"> for LCP
      // (Phase 13 Plan 5 Task 6). If auth rehydrates to 'authed' BEFORE Alpine.effect
      // first invokes openAuthWall(), wallEl stays null and the static element is
      // never decorated nor removed — leaving it covering the app shell with the
      // bare COUNTERFLUX h1 visible and no other interaction possible.
      // Belt-and-braces: also strip the static element when wallEl is null.
      const stale = typeof document !== 'undefined' ? document.getElementById('cf-auth-wall') : null;
      if (stale) stale.remove();
    }
    ```

    Append a regression test to `tests/auth-wall.test.js` inside the existing `describe('auth-wall — closeAuthWall', …)` block:

    ```js
    test('closeAuthWall removes the static #cf-auth-wall from index.html when wallEl is null', () => {
      const stale = document.createElement('div');
      stale.id = 'cf-auth-wall';
      stale.innerHTML = '<h1 class="cf-auth-wall-title">COUNTERFLUX</h1>';
      document.body.appendChild(stale);
      expect(document.getElementById('cf-auth-wall')).toBeTruthy();

      closeAuthWall();
      expect(document.getElementById('cf-auth-wall')).toBeNull();
    });
    ```
