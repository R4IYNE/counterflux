<objective>
Build the Scryfall bulk data pipeline: a Web Worker that downloads ~300MB of Default Cards JSON, stream-parses it with @streamparser/json-whatwg, and batch-inserts cards into Dexie.js IndexedDB — all without crossing the Worker boundary with card data. Wire this to an Alpine.js store for progress tracking and a blocking splash screen for first-load UX.

Purpose: This is the hardest technical challenge in Phase 1 — streaming 300MB of JSON without crashing the browser. Everything downstream (search, autocomplete, card display) depends on this data being in IndexedDB.
Output: Working bulk data pipeline with splash screen, daily refresh logic, and progress feedback.

       - Import Dexie and create an IDENTICAL schema to src/db/schema.js (same `db.version(1).stores({...})` — Dexie in the Worker needs the same schema declaration to access the shared IndexedDB database named 'counterflux').
       - Import `JSONParser` from `@streamparser/json-whatwg`.
       - Listen for `self.onmessage`:
         - `type: 'start'` with `{ downloadUri, expectedSize }`:
           1. `fetch(downloadUri)` with `{ headers: { 'User-Agent': 'Counterflux/1.0 (MTG collection manager)' } }`
           2. Get `response.body` as ReadableStream
           3. Read chunks with `reader.read()` loop
           4. Track `downloaded` bytes (sum of `value.byteLength`)
           5. Pipe bytes through JSONParser configured with `paths: ['$.*']` and `keepStack: false`
           6. Accumulate parsed card objects in a batch array
           7. When batch reaches 1000 cards: `await db.cards.bulkPut(batch)`, reset batch, post progress
           8. Post `{ type: 'progress', downloaded, total: expectedSize, parsed: totalParsed }` every 1000 cards
           9. On stream end: flush remaining batch with `bulkPut`, post `{ type: 'complete', totalParsed }`
           10. On error: post `{ type: 'error', message: error.message }`
         - `type: 'check'` with `{ cachedUpdatedAt }`:
           1. Fetch Scryfall bulk-data metadata endpoint
           2. Compare `updated_at` with `cachedUpdatedAt`
           3. Post `{ type: 'check-result', needsRefresh, downloadUri, updatedAt, expectedSize: data.size }`

       CRITICAL per RESEARCH.md Pitfall 2: Dexie runs INSIDE the Worker. Card data never crosses the Worker boundary via postMessage. Only progress numbers and control messages cross. This avoids serializing 300MB.

       CRITICAL per RESEARCH.md Pitfall 3: Never cache the `download_uri`. Always fetch fresh from `/bulk-data/default-cards`.

       CRITICAL per RESEARCH.md Pitfall 4: Use `expectedSize` from bulk-data metadata (the `size` field), NOT `Content-Length` header (may be missing due to chunked transfer).

    1. Create `src/stores/bulkdata.js` — Alpine store for bulk data state:
       ```javascript
       import Alpine from 'alpinejs';
       import { getBulkMeta, setBulkMeta } from '../db/schema.js';
       import { requestPersistentStorage } from '../utils/storage.js';

       export function initBulkDataStore() {
         Alpine.store('bulkdata', {
           status: 'idle',       // idle | checking | downloading | parsing | ready | error
           downloaded: 0,        // bytes downloaded
           total: 0,             // expected total bytes (from Scryfall size field)
           parsed: 0,            // cards parsed so far
           totalCards: 0,        // total cards when complete
           error: null,          // error message string
           updatedAt: null,      // ISO timestamp of cached data

           get progress() {
             if (this.total === 0) return 0;
             return Math.round((this.downloaded / this.total) * 100);
           },

           get downloadedMB() {
             return (this.downloaded / 1048576).toFixed(1);
           },

           get totalMB() {
             return (this.total / 1048576).toFixed(0);
           },

           get isReady() {
             return this.status === 'ready';
           }
         });
       }
       ```
