# Plan 14-05: Supabase column parity with Dexie client payloads

## Context

Live UAT of Plan 14-01 (Issue A fix — `payload.user_id = currentUserId` before Supabase upsert) surfaced a **second latent Phase 11 bug**: the `counterflux.*` tables on Supabase are missing columns that the local Dexie v8 stores write.

### How it surfaced
1. Plan 14-01 landed the spread-and-stamp fix. `user_id` now reaches the server.
2. User signed into huxley Supabase, added a Lightning Bolt.
3. Instead of landing cleanly, the sync engine reported "848 changes failed to sync" — every pending `deck_cards` queue entry dead-lettered.
4. Every failed row's `sync_conflicts.error_code` = `PGRST204`, `error_message` = "Could not find the 'sort_order' column of 'deck_cards' in the schema cache".
5. The 14-01 fix was genuinely working — payloads reached the server fully stamped. They were rejected by PostgREST at the schema-cache layer, not at the RLS/NOT-NULL layer. That's a drift between local schema and server schema, not a user_id problem.

### Why it was latent through Phase 11

Phase 11 happy-path tests (and the initial Phase 11 live sanity checks) only exercised the columns that the auth-foundation migration modelled — `id, user_id, deck_id, scryfall_id, quantity, category, is_commander, updated_at, synced_at`. The store write paths add more columns than that:

| Table | Client writes | Server column (pre-fix) |
|-------|---------------|-------------------------|
| `collection` | `added_at` (ISO) | missing |
| `decks` | `deck_size, partner_id, companion_id, color_identity, tags, created_at` | missing |
| `deck_cards` | `tags, sort_order` | missing |
| `games` | `turn_count, elimination_order, active_player_index` | missing |
| `watchlist` | `added_at, alert_type, alert_threshold, last_alerted_at` | missing |
| `profile` | — (no client writes yet) | no drift |

`deck_cards.sort_order` happened to be the first PGRST204 offender because `sort_order` is in every `deck_cards` payload — every deckbuilder drag fires the write. Other tables would have failed on their respective missing columns once their first write hit the server.

### Scope

Additive-only schema fix. Adds all 16 missing columns across 5 tables. No drops, no type changes, no data writes. Nullable with sensible defaults where the client expects one. `ADD COLUMN IF NOT EXISTS` makes the migration idempotent.
