<objective>
Close audit Issue A — the critical latent sync-push bug where `payload.user_id` reaches Supabase as `null` / `undefined` because the push path in `src/services/sync-engine.js` never stamps the current user's id before calling `.upsert(rows)`. The Supabase `counterflux.*` tables declare `user_id uuid NOT NULL` with no `DEFAULT auth.uid()` (per audit evidence), so the first new write after sign-in dead-letters with SQLSTATE 23502 (classified permanent by `classifyError()` at line ~313) and the user's notification bell lights up for every local edit.

Purpose: Ship Option A of the audit's three fix options — a surgical 1-line stamp at the batch-upsert site. Option A is recommended because it is minimal churn, single source of truth, no SQL migration needed, and Phase 10's `_currentUserId()` helper is already in scope at the call site (it's used 3 lines above to filter the queue).

Output: One modified source file, one new Vitest test, one human-verify UAT checkpoint, and a dated annotation on the Phase 11 HUMAN-UAT live-Supabase anchor.

Closes: SYNC-03 (push wiring), AUTH-06 (user_id enforcement end-to-end — currently client leaves it null, now stamped at the seam).

export async function flushQueue() {
  if (_authStatus() !== 'authed') return;
  if (typeof navigator !== 'undefined' && navigator.onLine === false) return;

  const currentUserId = _currentUserId();
  if (!currentUserId) return; // anonymous or missing user_id — blocks flush

  // ... dedup logic builds `latestByRow` Map ...

  if (latestByRow.size > 0) {
    const rows = Array.from(latestByRow.values()).map(e => e.payload);
    // ^^^ THIS IS THE SEAM — `e.payload` comes from sync_queue rows that captured
    //     the row snapshot via JSON.parse(JSON.stringify(obj)) in the Dexie hook,
    //     which preserves user_id: null / undefined from the source row.
    const { error } = await supabase.schema('counterflux').from(tableName).upsert(rows);

}
