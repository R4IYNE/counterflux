## Critical Gap: Issue A — Cloud Sync Push Will Reject

### The chain

1. `src/stores/collection.js:446` hard-codes `user_id: null` on new row inserts.
2. `src/stores/deck.js` + `src/stores/game.js` never set `user_id` — row enters Dexie with `user_id: undefined`.
3. `src/services/sync-engine.js:216-237` (`installSyncHooks` → creating hook) snapshots the row via `JSON.parse(JSON.stringify(obj))` → `null`/`undefined` on user_id survives into the sync_queue entry's `payload` field.
4. `src/services/sync-engine.js:410-411` (flushQueue) maps `rows = latestByRow.values().map(e => e.payload)` and calls `supabase.schema('counterflux').from(tableName).upsert(rows)` — **no user_id backfill**.
5. `supabase/migrations/20260417_counterflux_auth_foundation.sql` declares:
   ```sql
   user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
   ```
   — no `DEFAULT auth.uid()`, no BEFORE INSERT trigger. Upsert returns SQLSTATE `23502` (not_null_violation).
6. `classifyError(err)` at `sync-engine.js:312` matches `/^23\d{3}$/` → classified `permanent` → the row dead-letters into `sync_conflicts`.
7. User sees notification bell light up with "N operations failed" for every local write after sign-in.

### Why it hasn't been observed

- Phase 11 HUMAN-UAT `status: pending` — the live sync test was deferred and never run.
- The current user (James) may have always been signed-in since Phase 10 completed; if they have not created new rows since signing in, the sync queue stays empty and nothing tries to push.
- Reconciliation's `populated-remote → empty-local` branch is a PULL, not a push, so sign-in on a fresh device doesn't trigger the break either.

### Fix options (user picks)

| Option | Scope | Trade-off |
|--------|-------|-----------|
| **A. 1-line flushQueue fix** | Add `e.payload.user_id = currentUserId` before the upsert at `sync-engine.js:411` | Minimal churn. Single source of truth. Doesn't require SQL migration. |
| **B. Supabase schema migration** | Add `ALTER COLUMN user_id SET DEFAULT auth.uid()` on all 6 synced tables | Server-side guarantee. But UPSERT still sends `null` and may overwrite server value — needs test. Requires coordinated deploy. |
| **C. Stamp at row-creation (11+ files)** | Set `user_id: auth.user?.id` in every store's add call | Cleanest conceptually. Largest diff. Race condition if user signs in mid-session — some rows stamped, some not. |

**Recommended: Option A.** One surgical change + a live UAT to confirm. If the UAT passes, close the gap. If it reveals a deeper issue, escalate to B.
